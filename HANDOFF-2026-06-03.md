## Don't-repeat list — scars added to CLAUDE.md tonight

(Future Claude sessions reading CLAUDE.md will see these.)

### Date strings across system boundaries — must combine date + time + TZ

trustEd stores date and time in separate columns. Outbound code that crosses a boundary MUST use `_toCentralTimestamp(date, time)` — never `.toISOString().slice(0, 10)` direct to a fetch body. Always pass through the canonical helper `_cycleToCanonicalTimestamps(cycle)` if working with cycle data.

Added scar: "Date strings across system boundaries — must combine date + time + TZ" in CLAUDE.md anti-patterns.

### Bridge idempotency must UPDATE on re-push, not no-op

If bedrock-vote (or any external receiver) has its own stored copy of trustEd data, the bridge's "already exists" path must UPDATE high-level fields with the new payload. Otherwise trustEd's source-of-truth changes never flow through, and operators end up running SQL to fix data drift.

(Not yet in CLAUDE.md — add when the bedrock-vote fix is committed.)

### Mailing-delta apply was silently lying

The success popup said "✓ Applied. Mailing: 6. Errors: 14" — implying the apply worked. Actually 0 mailings were written; the 14 errors were CHECK constraint violations rejecting every write. The error count was a soft hint, not a STOP signal.

**Lesson:** if `errors > 0` in any "apply" / "ingest" / "sync" / "mark" endpoint response, the UI should not say "✓ Applied." Should say "⚠️ Partial: N applied, M errored." Show first 3 error messages inline.

Fixed in the alert text for the mailing-delta panel; pattern should be audited across other apply endpoints in next session.
